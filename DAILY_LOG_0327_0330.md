**Experiment 3: Tool-Use Ablation (Misviz real, 50 charts)**

Three-condition comparison results:

| Condition | n | Binary F1 | Precision | Recall | Avg Time |
|-----------|---|-----------|-----------|--------|----------|
| VLM-only (Member B) | 271 | 83.0% | 78.2% | 88.3% | ~5s |
| Pipeline (OCR+Rules → single VLM) | 28 | **92.3%** | **100%** | 85.7% | ~60s |
| Agentic (multi-turn tool-use) | 10 | 82.4% | 100% | 70.0% | ~100s |

**Experiment 4: Real SEC Filing Chart Analysis (single case)**
- FXLV `fxlv-20211231_g1.jpg`: T2 Pipeline detected 8 findings (truncated axis HIGH 95%, 6 completeness issues)
- Validated T2+T3 complementarity: T2 catches chart-level visual defects, T3 catches filing-level Non-GAAP pairing issues

### Key Findings

1. **Pipeline mode is optimal for small VLMs**: Pre-computing OCR + rule evidence and injecting into a single VLM prompt outperforms both VLM-only and agentic multi-turn (F1: 92.3% vs 83.0% vs 82.4%)

2. **Agentic multi-turn hurts small models**: Haiku 4.5 degrades in multi-turn tool-use — uses OCR results as noise, FPs increase, latency 20x higher

3. **Evaluator naming bug fixed**: `truncated_axis` (underscore) vs `truncated axis` (space) caused per-type F1 to be severely underreported. After normalization, truncated axis F1 jumped from 0% to 75%

4. **PaddleOCR memory leak**: Cannot run >10-12 charts in a single process. Solved with subprocess isolation per chart

5. **Large filing (>1MB HTML) causes 400 errors**: VLM context limit exceeded. Need chunking strategy for 10-K filings
